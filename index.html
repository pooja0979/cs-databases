üí° Why Relational Databases?

The Solution!

Relational Database: A database that stores data in multiple related tables instead of one big table

Key Idea: Split Data into Separate Tables!

From Flat File ‚Üí Relational Database

Benefits of Separating Tables:

  • ‚úì Each piece of data stored ONCE - no redundancy!
  • ‚úì Easy to update - change in one place only
  • ‚úì No anomalies - can insert, update, delete freely
  • ‚úì Better organized - logical structure

2. Primary Key (PK)

Primary Key: A field (or combination of fields) that uniquely identifies each record in a table

Student Table

StudentID (PK) Name Email
S001 Alice Chen alice@school.com
S002 Bob Smith bob@school.com
S003 Alice Wang alice.w@school.com

Why StudentID is the Primary Key:

  • ‚úì Each value is UNIQUE (no duplicates)
  • ‚úì Never NULL (empty)
  • ‚úì Never changes
  • ‚úó Name cannot be PK - we have two "Alice"s!
‚ö° Primary Key Rules:
  • Must be UNIQUE for each record
  • Cannot be NULL (empty)
  • Should never change
  • Keep it simple (preferably one field)

Foreign Key: A field in one table that refers to the Primary Key in another table

4. Composite Key

Composite Key: When TWO or MORE fields are combined to uniquely identify a record

‚Üí Used when no single field can be unique on its own

Example: Student-Course Registration

Registration Table
Composite Key (PK) Semester Grade

üí° Why Composite Key?
  • StudentID alone is NOT unique (S001 appears twice)
  • CourseCode alone is NOT unique (CS101 appears twice)
  • But the COMBINATION (S001 + CS101) IS unique! ‚úì

5. Concatenated Key

Concatenated Key: Similar to composite key - multiple fields joined together to form a unique identifier

Note: Often used interchangeably with "composite key" in practice

A library may have multiple copies of the same book:

Concatenated Key = BookISBN + CopyNumber

üîó Relational Database Features (Part 4)

6. Relationships

Relationship: How entities (tables) are connected to each other

Types of Relationships:

a) One-to-One (1:1)

One record in Table A relates to ONE record in Table B

Example: Each student has ONE passport, each passport belongs to ONE student

b) One-to-Many (1:M) - Most Common! ‚≠ê

One record in Table A relates to MANY records in Table B

Example: One teacher teaches MANY students, but each student has one homeroom teacher

c) Many-to-Many (M:N)

Many records in Table A relate to MANY records in Table B

Example: Many students take many courses. We need a junction table to connect them!

7. Cardinality

Cardinality: The maximum number of times an entity can be associated with another entity

‚Üí Answers: "How many at MOST?"

Cardinality Examples:

Relationship Cardinality Meaning
Teacher ‚Üí Student 1:M One teacher can have MANY students
Student ‚Üí Course M:N Many students, many courses
Country ‚Üí Capital 1:1 One country, one capital city

8. Modality (Participation)

Modality: The minimum number of times an entity must be associated with another entity

‚Üí Answers: "Is participation required?"

  • Mandatory (1): Must participate - cannot be zero
  • Optional (0): May or may not participate

Modality Notation (Crow's Foot)

üéØ Remember:
  • Cardinality = Maximum (1, M, N)
  • Modality = Minimum (0 or 1)

‚úÖ Benefits of Relational Databases

Why are Relational Databases so Popular?

1. Data Integrity & Consistency

  • Primary keys ensure each record is unique
  • Foreign keys maintain referential integrity
  • No update/insert/delete anomalies!
  • Data validation rules can be enforced

2. Reduced Data Redundancy

  • Each piece of data stored only ONCE
  • Changes made in one place automatically reflect everywhere
  • Saves storage space
  • Easier to maintain data accuracy

Example: Teacher's name stored once in Teachers table, referenced by TeacherID in multiple courses

3. Data Independence

  • Can change database structure without affecting applications
  • Physical storage separate from logical structure
  • Flexibility in database design

4. Flexibility & Scalability

  • Easy to add new tables and relationships
  • Can handle growing amounts of data
  • Multiple users can access simultaneously
  • Can query data in many different ways

5. Security & Access Control

  • User permissions can be set at table/field level
  • Sensitive data can be protected
  • Audit trails to track changes

Example: Students can view grades but cannot edit them; only teachers can update grades

‚úÖ Benefits (continued) & ‚ö†Ô∏è Limitations

6. Powerful Query Language (SQL)

  • Standardized language (SQL) used across all relational databases
  • Can perform complex queries joining multiple tables
  • Supports sorting, filtering, aggregation
  • Easier for humans to read and write

Example SQL: SELECT Name FROM Students WHERE Grade > 90;

7. ACID Properties (Reliability)

  • Atomicity: Transactions complete fully or not at all
  • Consistency: Data remains valid after transactions
  • Isolation: Concurrent transactions don't interfere
  • Durability: Committed data is permanently saved

‚ö†Ô∏è Limitations of Relational Databases

1. Complexity

  • Requires careful planning and design (normalization)
  • Complex relationships can be difficult to model
  • Need to understand SQL and database concepts
  • Initial setup time-consuming

2. Performance Issues with Large Data

  • Joining many tables can be slow
  • Performance degrades with billions of records
  • Complex queries may take significant time
  • Indexing required for optimization (adds overhead)

3. Scalability Challenges

  • Vertical scaling (more powerful hardware) is expensive
  • Horizontal scaling (multiple servers) is difficult
  • Maintaining consistency across distributed systems is hard
Page 10

‚ö†Ô∏è Limitations (continued)

4. Rigid Schema

  • Must define structure (tables, fields, types) upfront
  • Changing structure later can be difficult and disruptive
  • All records must fit the same schema
  • Not ideal for unstructured or rapidly changing data

Example: If you want to add a new field "MiddleName" to Students table, you must modify the schema and potentially migrate existing data

5. Not Ideal for Certain Data Types

  • Hierarchical data (like XML, JSON) doesn't fit well in tables
  • Graph data (social networks) requires many joins
  • Large binary objects (videos, images) are inefficient

6. Cost

  • Commercial databases (Oracle, MS SQL Server) can be expensive
  • Requires skilled database administrators (DBAs)
  • Hardware and maintenance costs

7. Object-Relational Impedance Mismatch

  • Objects in programming languages don't map perfectly to tables
  • Need ORM (Object-Relational Mapping) tools
  • Can lead to complex code

Real-World Database Examples

Popular Relational Database Systems:

  • MySQL - Open source, widely used for web applications
  • PostgreSQL - Advanced open source, strong data integrity
  • Oracle Database - Enterprise-level, very powerful
  • Microsoft SQL Server - Integrates with Microsoft ecosystem
  • SQLite - Lightweight, used in mobile apps

School Library System

Design a relational database for a school library. Consider:

  • Students can borrow multiple books
  • Books can be borrowed by different students (but one at a time)
  • Track borrowing date and return date
  • Books have ISBN, title, author, genre

Proposed Solution:

üéØ Design Rationale:

  • Primary Keys: StudentID, ISBN, BorrowingID
  • Foreign Keys: Borrowing table references both Students and Books
  • Benefits: No redundancy, can track borrowing history, easy to query
